Steps to follow before requesting a VeriSign Server ID
Due to numerous variations within Microsoft's IIS software platform and its service packs, a number of customers have been encountering the error "Invalid Password" when trying to install their Server ID. This error is caused by a slight misconfiguration in the IIS Key Manager software. This error can be prevented by carefully following the steps outlined below.
IF YOU DO NOT FOLLOW THESE STEPS, THERE IS A HIGH PROBABLILITY THAT YOUR CERTIFICATE WILL NOT WORK PROPERLY, AND YOUR INSTALLATION WILL BE DELAYED.
  1. If you are located in the United States or Canada, it is recommended that you apply Service Pack 3 for Windows NT, 128 bit version. This is available from Microsoft on CD ROM, or if you have a registered IP address, you may download the file from Microsoft's World Wide Web site. You can obtain the 128 bit version of Service Pack 3 online.
  2. Create a test Certificate Signing Request in Key Manager and register for a test Certificate with Verisign. Make sure you use a password that is 7 characters in length or less!.

How to use KEYGEN.EXE and SETKEY instead of Key Manager NOTE: If you are unable to use the following utilities, please contact Microsoft Technical Support at 1-800-936-3500.
  1. Open a command prompt
  2. Change directories to the \system32\inetsrv directory
  3. Enter the following:
    keygen -bits <512 or 1024> "C=, O=, OU=, S=, L=, CN=" Example:
    keygen -bits 512 secret privkey.txt certreq.req "C=US, O=my company, OU=my unit,S=Washington, L=Seattle, CN=www.walleye.com "

    NOTE: "512 " can ONLY be "1024" if you are using 128 bit encryption "secret" is the password (should be 7 characters or less) "privkey.txt" is a name you give to the Private Key (can be anything, don't use spaces) "certreq.req" is a name you give to the request file (can be anything, don't use spaces)
  4. You should see something like the following if it works: "SSL Key generation utility, Version 1.0 Copyright (c) 1995 Microsoft Corporation Generating key pair of length 512 bits... Completed.
  5. If you are successful, cut and paste the request file certreq.req in the VeriSign enrollment form
  6. When you get the Certificate back from Verisign, do the following:
    • Open a command prompt
    • Change directories to the \system32\inetsrv directory
    • Enter the following:
      setkey secret privkey.txt newcert.cer www.x.com

      Note:
      "secret" is the password from the KEYGEN process
      "privkey.txt" is the Private Key from the KEYGEN process
      "newcert.cer " is the Certificate from Verisign.
      "www.x.com " is the Common Name from the KEYGEN process
      You should see the following if the process works:
      "SSL Key set utility, Version 1.0
      Copyright (c) 1995 Microsoft Corporation
      Successfully set the SSL keys for www.x.com in the registry! "
If using Keygen and Setkey result in an error, the errors are due to system configuration issues Please contact Microsoft Techinical Support at 1-800-936-5900. Be advised that this is paid support.

Generating a Key Pair and CSR for a Microsoft Server IMPORTANT NOTICE: If you are using a Microsoft Internet Information Server, you must use a password that contains eight characters or less when generating your CSR. To generate a public and private key pair and CSR for a Microsoft Internet Information Server 2.0:
  1. Launch the Microsoft Internet Service Manager.
  2. Select Key Manager from the Microsoft Internet Server submenu. (Or click on the key manager icon in the toolbar.)
  3. Select Create New Key from the Key menu. The Create New Key and Certificate Request dialog box displays.Do not enter commas in any of these fields. Commas are interpreted as the end of the field and an invalid CSR will be generated.
  4. In the Key Name field, specify a name for the key you are creating.
  5. In the Password field, specify a password to protect your private key. This password should be no more than eight characters long and contain at least one nonalphabetical character, such as a number or punctuation mark. Remember, the security of your server is only as good as the security of the key-pair file and its password. You will need this password to install your Secure Server ID. If you lose your private key or forget the password, you will not be able to install the ID and will need to request and purchase a new ID from VeriSign. encrypt the private key.
  6. By default, Key Manager generates a key pair 1024 bits long. In international servers, the size of the key you create is 512 bits long. For our international customers who are interested in obtaining a key with greater encryption strength, go to http://www.verisign.com/globalserver.html.
  7. In the Organization field, enter the name of your organization. This is the organization name you wrote down in Enrollment Step 1. The name you enter must be identical to the name listed as the owner of the domain name. VeriSign can only issue a Secure Server ID to the organization that owns the specified domain name. If you are unsure of the correct organization name to enter, go back to Step 1: Confirm Domain Name to look up the domain name registration.
  8. In the Organizational Unit field, enter the name of the department or unit that will be using the ID. For example, enter the name of your department or business unit. This field is not required.
  9. In the Common Name field, enter your web site's fully-qualified name. For example, www.company.com. Do not include the protocol, such as HTTP:// or FTP://.
  10. In the Country field, enter the two-letter code for your country. You must use the correct ISO country code, other abbreviations will not be recognized. For example, the correct code for the United Kingdom is GB, not UK.
  11. In the State/Province field, enter the full name of your state or province. Do not abbreviate.
  12. In the Locality field, enter the name of your city, town, or other locality.
  13. In the Request File field, enter a name for the CSR file. The CSR will look something like this:

  15. When prompted, enter the password you entered in Step 5: Complete Application, and click OK. An icon is displayed while the key is being generated.   16. Click OK to dismiss the information window.
  17. Select Commit Changes Now from the Servers menu to save your new key.
  18. When asked if you want to commit all changes now, click OK. Your key will be listed in the Key Manager window under the name of the computer for which you created the key. By default, the key is generated on the local computer. The CSR is saved in the file that specified as the Request File. If you do not commit the changes, you will not be able to install your Secure Server ID.
  19. Back up your private key file (*.key ) and CSR file (key.req on a floppy disk and store the disk in a secure location. If you lose your private key or forget the password, you will not be able to install your Secure Server ID and will need to request and purchase a new one from VeriSign.
  20. Go back to the enrollment pages and follow the instructions in Enrollment Step 4: Submit CSR to submit your CSR.
  return to the top Generating a Key Pair and CSR For a Remote Server The Key Manager tool (keyring.exe) with IIS 2.0 enables you to generate a key pair and install a Secure Server ID on a remote server. WARNING: Transferring a key pair (*.key file) from one computer to another over a network can expose it to snooping because the key pairs are not encrypted when sent across the network. If a private key is compromised, it can compromise the site's SSL identification (as stated in the site certificate). The compromise of a private key means that all subsequent SSL communications to and from that computer are potentially subject to interception and decryption. To protect your private key, which is paramount to SSL security, you should either:
  1. Generate the key locally on the computer where it is to be installed.
  2. Transfer the key from the remote machine by saving it to a floppy disk and hand-carrying it to the target machine and installing it from the floppy.
  3. Transfer the private key across the network only if you are in a secure environment and it is a trusted network.
